# 루프 엔지니어링

코딩 에이전트와 일하는 방식이 **매 턴 직접 프롬프트**에서, 에이전트를 대신 찌르는 **시스템을 설계**하는 것으로 이동하는 패턴.

개념 진화: 프롬프트 → 컨텍스트 → harness → **루프** (harness engineering·factory model **상위**).

| 인용 | 내용 |
|------|------|
| Peter Steinberger | *"에이전트에 프롬프트하는 루프를 설계하라"* |
| Boris Cherny (Claude Code) | *"내 일은 루프를 작성하는 것"* |

**루프** = 목적을 정의하면 AI가 완료될 때까지 반복하는 **재귀적 목표(recursive goal)**.

## 5단계 사이클 (DPEVI)

모든 루프는 규모와 무관하게 동일한 사이클 ([[출처/루프 엔지니어링 학습 자료 PyTorchKR]]):

| 단계 | 역할 |
|------|------|
| **Discover** | 목표 이해·정보 탐색 |
| **Plan** | 작업 계획 수립 |
| **Execute** | 코드·콘텐츠 등 실제 수행 |
| **Verify** | 목표·기준 충족 검증 |
| **Iterate** | 실패 시 1단계로 — 통과 시 완료 |

골격: **Goal → Action → Check → Fix → Repeat until done**.

## 5+1 구성요소

Claude Code와 Codex 모두 **다섯 가지 + 메모리**를 제품 기능으로 제공 (1년 전 bash 더미 수준).

| # | 요소 | 역할 |
|---|------|------|
| 1 | **Automations** | 스케줄 발동, triage — 루프의 심장 박동 |
| 2 | **Worktrees** | 병렬 에이전트의 파일 충돌 방지 |
| 3 | **Skills** | 프로젝트 지식·의도를 `SKILL.md`로 외부화 |
| 4 | **Plugins·connectors** | MCP로 이슈·DB·Slack 등 실환경 연동 |
| 5 | **Sub-agents** | maker / checker 분리 |
| 6 | **Memory** | markdown·Linear·state file — **디스크**에 상태 (모델은 실행 사이 잊음) |

### Automations & `/goal`

- Claude Code: `/loop`(주기 재실행), hooks, cron, GitHub Actions
- Codex: Automations 탭, Triage 인박스, `$skill-name`
- **`/goal`**: 검증 가능 정지 조건이 참이 될 때까지 반복; **별도 작은 모델**이 완료 검사 → 코드 작성 에이전트가 채점자가 되지 않음

### Worktrees

git worktree로 repo 공유·체크아웃 분리. 상한은 도구가 아니라 **리뷰 대역폭**.

### Skills

intent debt 방지 — skill 없으면 매 사이클 0에서 재유도, 있으면 복리적 누적. skill=저작, plugin=배포.  
프로젝트 문서 triad: `VISION.md`(성공 정의), `ARCHITECTURE.md`(스택·구조), `RULES.md`(금지 사항) — [[출처/루프 엔지니어링 학습 자료 PyTorchKR]].

### Sub-agents

자기 숙제 채점은 관대 → 탐색·구현·검증 분담. [[Claude Managed Agents]] Outcomes grader, `/goal` checker와 동일 원리.

## 규모·유형

### 단일 vs 플릿

| | 단일 에이전트 루프 | 플릿 루프 |
|---|-------------------|-----------|
| 구조 | 한 에이전트가 전체 DPEVI | Orchestrator → Specialist → Subagent **트리** |
| 적합 | 집중·단순·제한 범위 | 대규모·다단계 프로젝트 |
| 품질 | self-improve | **Eval Gate** |

### Open vs Closed

| | Open Loop | Closed Loop |
|---|-----------|-------------|
| 특성 | 탐색적·넓은 행동 공간 | 사람이 E2E 경로·평가·escalation 설계 |
| 비용 | 토큰 폭발 (Steinberger식) | 일반 예산으로 운영 가능 |
| 실무 | 예산 무제한 소수 | **오늘의 기본 선택** — 타이트 클로즈드 → 점진적 개방 |

## 대표 루프 패턴

[[출처/루프 엔지니어링 학습 자료 PyTorchKR]] — Coding · Research · Content · Sales outreach 4종. Coding 예: VISION+ARCHITECTURE 읽기 → 계획 → 편집 → 테스트 → 실패 시 수정 루프.

## Self-correction loop (inner)

평가 기준(**goal**, **rubric**)을 환경에 주입해 **hillclimb**.

```
실행 → 피드백 수집 → 자가 수정 → (rubric 충족까지 반복)
```

| 구성요소 | 역할 |
|----------|------|
| Rubric / goal | 체크 가능한 성공 기준 |
| Verifier sub-agent | self-critique보다 우수 — 독립 context ([[출처/Fable 5 루프 설계 GeekNews]]) |

### Parameter Golf 관찰 ([[Claude Fable 5]] vs Opus 4.7)

- Fable 5: 구조적 변경·회복 — **~6×** 파이프라인 개선
- Opus 4.7: 스칼라 튜닝 반복

## Memory loop (outer)

세션 간 memory 검색·재사용. CMA mounted filesystem 등.

### fail → investigate → verify → distill → consult

| 모델 | 도달 | 검증 커버리지 |
|------|------|---------------|
| Sonnet 4.6 | ~fail | — |
| Opus 4.7 | ~verify | median **~17%** |
| Fable 5 | ~distill | 최대 **73%** |

## 예시 합성 루프

매일 automation → triage skill → CI·이슈·커밋 스캔 → **state file** 기록 → 항목별 worktree → sub-agent 초안 + 리뷰 sub-agent → connector(PR·티켓) → 미처리 triage 인박스. **한 번 설계, 반복 프롬프트 없음.**

## 루프가 대신 못 하는 것

1. **검증** — 무인 루프 = 무인 실수; "done"은 증명이 아니라 주장
2. **이해** — comprehension debt; 루프 산출물을 읽지 않으면 간극 확대
3. **판단** — cognitive surrender; 같은 루프가 치료제이자 가속제

## 토큰 비용

설계 난이도보다 **토큰**이 실무 장벽 ([[출처/루프 엔지니어링 학습 자료 PyTorchKR]] 추정):

| 시나리오 | 토큰 |
|----------|------|
| 단일·중간 코딩 | ~50k–200k |
| 플릿 (orchestrator + 전문가 3) | ~500k–2M |
| 매일 스케줄 | 주간 수백만 |

재시도·sub-agent·검증마다 누적. token rich → Open Loop; token poor → **Closed Loop**·저가 장컨텍스트 모델(DeepSeek V4 등) 검토.

## 설계 원칙

- Rubric·종료 조건을 **체크 가능**하게
- Grader를 작업 에이전트와 **분리**
- 토큰 비용·루프 유형(Open/Closed)을 예산에 맞게
- ***Build the loop, stay the engineer*** — 레버리지 지점 이동, 직접 프롬프트와 **균형**
- *하나의 신뢰할 수 있는 루프 > 천 개의 완벽한 프롬프트*
