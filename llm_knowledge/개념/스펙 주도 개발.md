# 스펙 주도 개발

**Spec-Driven Development (SDD)** — LLM 시대 AI 코딩에서 **코드 전에 명세(spec)** 를 중심에 두는 워크플로. 명세가 개발자와 AI의 **single source of truth**.

GitHub 정의: *소프트웨어 유지보수 = 명세의 진화; 공통 언어가 더 높은 수준으로 올라가고 코드는 최종 단계.*

## 세 구현 수준

| 수준 | 개발자 | 명세 역할 |
|------|--------|-----------|
| **spec-first** | 명세 작성 → AI로 구현 | 작업 시작 전 PRD급 입력 |
| **spec-anchored** | 기능 수명 동안 명세 유지 | 진화·유지보수의 기준 |
| **spec-as-source** | **명세만** 편집 | 코드는 `GENERATED FROM SPEC`, 직접 수정 안 함 |

모든 SDD는 spec-first에서 출발. anchored/as-source는 도구·팀 성숙도에 따라 선택.

## spec vs 전역 컨텍스트

| 산출물 | 범위 | 예 |
|--------|------|-----|
| **명세(spec)** | 특정 기능·변경 | user story, acceptance criteria, plan |
| **메모리 뱅크** | 전체 코드베이스 | constitution, steering, `AGENTS.md`, product.md |

혼동하면 기능 명세와 프로젝트 규칙이 뒤섞여 리뷰·에이전트 추적이 어려워짐.

## 대표 도구 (2025)

- [[Kiro]] — 요구사항 → 설계 → 작업 (3단, spec-first)

## 에이전틱 코딩에서의 동기

[[에이전틱 코딩]]은 생산성은 높지만 프로젝트 고도화 시 **무분별한 생성·환각·검토 시간 증가** ([[카카오페이 AI 플랫폼]]). SDD는 **스펙 불일치 시 직접 코드/MD 수정 대신 specify 재정의**로 single source of truth 유지. 개발자 역할: 코드 작성 → **스펙 검증·테스트 설계**.

## MDD와의 평행

**Model-Driven Development** — UML/DSL 명세 → 코드 생성기. 비즈니스 앱에서 실패(awkward abstraction, overhead).

| | MDD | SDD + LLM |
|--|-----|-----------|
| 명세 형식 | 파싱 가능 DSL | 자연어·마크다운 |
| 생성 | 맞춤 생성기 | LLM |
| 장점 | 구조·일관성 도구 지원 | 유연한 표현 |
| 위험 | rigidity | **비결정성** + rigidity(spec-as-source) |

spec-from-code 역사도 참고 — spec-driven 탐구 시 **양쪽 단점**(비유연·비결정)을 동시에 가질 수 있음.

## 한계 (실무 관찰)

1. **문제 크기**: 단일 워크플로로 호두·해머 (작은 버그에 16 AC)
2. **리뷰 UX**: 장황한 MD > 코드 diff
3. **에이전트 준수**: 지시·헌법·연구 결과 무시 또는 과잉 적용
4. **용어 확산**: "spec" ≈ "상세한 프롬프트"로 의미 희석
5. **대상 불명**: PM vs 개발자 역할, 그린필드 vs 레거시

spec-first **원칙**(명세 형태를 신중히 작성)은 [[Claude 프롬프트 엔지니어링]]·에이전트 설계와 겹치는 가치가 있음. "SDD 프레임워크 전체"는 아직 검증 중.
